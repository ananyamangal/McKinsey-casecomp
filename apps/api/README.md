# Halo API

FastAPI backend for Halo. **AI-free business logic** executed by a
**Workflow Engine**, with a **pluggable AI decision layer** and **mocked
integration adapters** behind clean interfaces.

## Run locally

```bash
cd apps/api
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Zero-config: defaults to SQLite (./halo.db). No services required.
python -m app.db.seed            # create schema + demo data
uvicorn app.main:app --reload    # http://localhost:8000/docs
```

Default login: `owner@apexmotors.in` / `password`.

### Use PostgreSQL instead

```bash
export DATABASE_URL="postgresql+psycopg2://halo:halo@localhost:5432/halo"
python -m app.db.seed
uvicorn app.main:app --reload
```

Copy `.env.example` → `.env` to customize anything (JWT secret, token TTLs,
CORS origins, Redis/Celery URLs).

## Data model (24 tables)

`Dealership`, `Role`, `User`, `Customer`, `Vehicle`, `Appointment`,
`WorkOrder`, `WorkOrderItem`, `ServiceBay`, `Technician`, `InventoryItem`,
`InventoryTransaction`, `Supplier`, `PurchaseOrder`, `PurchaseOrderItem`,
`Invoice`, `Payment`, `Notification`, `TelematicsEvent`, `AgentExecution`,
`WorkflowExecution`, `WorkflowStep`, `AuditLog` (+ `user_roles` association).

Conventions: String UUID PKs, `created_at`/`updated_at` mixins, Python enums via
`Enum(...)`, **money stored as integer minor units** (paise/cents), JSON columns
use JSONB on Postgres / JSON on SQLite.

## Workflow Engine

A persisted state machine over `WorkflowExecution` + `WorkflowStep` rows.

- `start(definition_key, context)` — create rows, run to completion/pause.
- approval gates pause at `WAITING_APPROVAL`; `approve(id)` releases them.
- `retry(id)` re-runs a failed step; `resume(id)` continues; `cancel(id)` stops.
- every transition publishes an event to the `EventBus`.
- step handlers are plain callables returning an output dict (merged into the
  shared context) or raising to fail.

## AI layer (pluggable)

Agents enrich context via the `ToolRegistry`, ask their injected
`DecisionEngine` whether to act, and return `ProposedAction`s. The
`agent_service` records each decision as an `AgentExecution` and — only if asked
— starts the proposed workflow through the engine. Replace `MockDecisionEngine`
with an `LLMDecisionEngine` (same interface) with zero changes elsewhere.

## Migrations (Alembic)

Wired to `Settings.DATABASE_URL` and `Base.metadata` (with SQLite batch mode and
`compare_type`). An initial autogenerated migration is included.

```bash
alembic revision --autogenerate -m "your change"
alembic upgrade head
```

For zero-config SQLite dev, tables are also auto-created on app startup and by
`python -m app.db.seed`.
